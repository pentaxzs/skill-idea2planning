# PRD 이해하기

# PRD는 무엇인가요?

### PRD(Product Requirements Document)는 무엇을 만들고, 어떻게 동작해야 하는지를 정의하는 문서입니다.

프로덕트의 요구사항, 정책, 예외처리, 사용자 흐름, 운영방식 등을 상세하게 정리하여
기획 / 디자인 / 개발 / QA / 운영 조직이 동일한 기준으로 제품을 만들 수 있도록 돕는 문서입니다.

쉽게 말하면:

* One-pager → “왜 이걸 해야 하는가”
* PRD → “그래서 실제로 어떻게 만들 것인가”

를 설명하는 문서입니다.

---

# 언제 / 왜 PRD를 쓰나요?

## 1. 실제 제작을 시작하기 전에 씁니다.

서비스를 개발하기 전:

* 어떤 기능이 필요한지
* 어떤 정책으로 동작하는지
* 어떤 예외상황이 있는지
* 어떤 데이터를 저장해야 하는지

를 명확하게 정의합니다.

---

## 2. 협업 기준을 맞추기 위해 씁니다.

같은 기능이라도:

* 기획자
* 디자이너
* 개발자
* QA
* 운영

가 서로 다르게 이해할 수 있습니다.

PRD는 모두가 동일한 기준으로 이해하도록 만드는 “공식 기준 문서” 역할을 합니다.

---

## 3. 정책/운영 리스크를 줄이기 위해 씁니다.

실제 서비스는 단순 화면 제작만으로 끝나지 않습니다.

예시:

* 결제 실패 시?
* 중복 가입 시?
* 지역 제한은?
* 미성년자 가입은?
* 해지 후 재가입은?
* 어뷰징 발생 시?

같은 예외상황까지 미리 정의해야 운영 리스크를 줄일 수 있습니다.

---

# One-pager와는 무엇이 다른가요?

| 구분    | One-pager | PRD           |
| ----- | --------- | ------------- |
| 목적    | 과제 설득     | 실제 제작         |
| 핵심    | Why 중심    | What / How 중심 |
| 상세도   | 간략        | 매우 상세         |
| 독자    | 의사결정자     | 실무자           |
| 작성 시점 | 과제 초기     | 착수 이후         |
| 포함 내용 | 문제/가치     | 정책/기능/예외처리    |
| 공수    | 비교적 적음    | 비교적 큼         |

---

# 좋은 PRD의 특징

# 1. 애매한 표현이 없음

❌ “적절하게 노출한다”

✅

* 최근 주문 이력이 있는 사용자에게만 노출
* 하루 최대 3회 노출
* 마지막 노출 후 24시간 이후 재노출 가능

좋은 PRD는 누구나 같은 방식으로 이해할 수 있어야 합니다.

---

# 2. 예외상황까지 정의되어 있음

기능보다 중요한 건 예외처리인 경우가 많습니다.

예시:

* 로그인 만료 상태
* 네트워크 실패
* 결제 실패
* 중복 요청
* API Timeout
* 권한 없음

실제 서비스 품질은 Edge Case 정의 수준에 따라 크게 달라집니다.

---

# 3. 정책과 UX가 함께 설명됨

좋은 PRD는 단순 정책문서가 아닙니다.

아래 내용이 함께 연결되어야 합니다.

* 왜 이런 정책이 필요한가
* 사용자에게 어떻게 보이는가
* 어떤 UX로 풀어내는가
* 운영상 어떤 의미가 있는가

---

# 4. KPI와 연결되어 있음

좋은 PRD는 기능 설명만 하지 않습니다.

반드시:

* 어떤 지표를 개선하려는지
* 성공 기준이 무엇인지
* 어떻게 측정할 것인지

까지 포함되어야 합니다.

예시:

* CTR +15%
* 가입전환율 +8%
* 리텐션 +5%
* 주문전환율 +2%p

---

# 5. 상태값(State)이 명확함

서비스는 대부분 상태(State) 기반으로 동작합니다.

예시:

| 상태        | 설명    |
| --------- | ----- |
| ACTIVE    | 정상 이용 |
| PENDING   | 승인 대기 |
| EXPIRED   | 만료    |
| CANCELLED | 해지    |

상태 정의가 애매하면:

* 개발
* 운영
* CS
* 데이터 분석

이 모두 꼬이기 시작합니다.

---

# PRD 작성 원칙

# 1. 기능보다 문제를 먼저 설명합니다.

❌ “추천 영역 추가”

✅

* 현재 추천 CTR 1.2%
* 반복 노출 피로 증가
* 개인화 정확도 낮음
* 따라서 추천 구조 개선 필요

기능은 문제 해결의 결과여야 합니다.

---

# 2. 숫자로 설명합니다.

❌ “사용률이 낮음”

✅

* 최근 30일 사용률 2.1%
* 신규 유저 사용률 0.8%
* 전월 대비 -18%

PRD는 가능한 한 정량 기반으로 작성합니다.

---

# 3. 정책 / UX / 운영을 함께 고려합니다.

실제 서비스는 아래가 동시에 연결됩니다.

| 영역  | 예시        |
| --- | --------- |
| 정책  | 가입 제한     |
| UX  | 가입 실패 메시지 |
| 운영  | CS 대응     |
| 데이터 | 로그 저장     |
| 개발  | API 처리    |

좋은 PRD는 이 흐름이 끊기지 않습니다.

---

# 4. 구현 가능한 수준까지 구체적으로 작성합니다.

PRD는 아이디어 문서가 아닙니다.

개발자가:

* API 설계
* DB 설계
* 상태 처리
* 예외 처리

를 시작할 수 있을 정도의 구체성이 필요합니다.

---

# 5. 리스크와 운영 이슈를 포함합니다.

실제 운영에서는 아래가 자주 발생합니다.

* 어뷰징
* 정책 충돌
* CS 증가
* 마케팅 오남용
* 데이터 누락
* 실험 오판

좋은 PRD는 리스크까지 미리 고려합니다.

---

# PRD의 일반적인 구성

# 1. 배경 및 문제정의

* 현재 상황
* 문제점
* 사용자 Pain Point
* 비즈니스 문제

---

# 2. 목표 및 KPI

* 프로젝트 목표
* 핵심 KPI
* 성공 기준
* 측정 방식

---

# 3. 사용자 및 정책 정의

* 대상 사용자
* 가입 조건
* 상태값
* 제한사항
* 예외정책

---

# 4. 기능 요구사항

* 기능 상세
* 유저 플로우
* 화면 정의
* API 요구사항
* 데이터 저장 정책

---

# 5. 운영 및 리스크

* 운영 정책
* CS 대응
* 로그 정책
* 어뷰징 대응
* QA 고려사항

---

# 좋은 PRD를 위한 체크리스트

## 문제 정의가 충분한가?

## KPI가 명확한가?

## 정책이 충돌하지 않는가?

## 상태값이 정의되어 있는가?

## 예외상황이 정의되어 있는가?

## 운영 가능한 구조인가?

## 개발 가능한 수준까지 구체적인가?

## QA 가능한 수준으로 정의되어 있는가?

---

# 추천 문서 분량

| 유형           | 권장 분량  |
| ------------ | ------ |
| 간단한 기능 개선    | 5~10p  |
| 일반 프로젝트      | 15~30p |
| 정책 중심 서비스    | 30~80p |
| 결제/구독/광고 시스템 | 50p+   |

---

# 추천 구성 비율

| 영역      | 권장 비중 |
| ------- | ----- |
| 문제정의    | 15%   |
| 정책 정의   | 30%   |
| 기능 요구사항 | 30%   |
| 예외처리    | 10%   |
| 운영/리스크  | 10%   |
| 기타      | 5%    |

---

# 마지막으로

좋은 PRD는 단순히 “문서를 잘 쓴 상태”가 아닙니다.

* 제품이 어떻게 동작해야 하는지
* 왜 그렇게 설계되었는지
* 운영 가능한지
* 리스크는 없는지

를 팀 전체가 명확하게 이해할 수 있는 상태를 만드는 것이 핵심입니다.

PRD는 문서 자체가 목적이 아니라,
좋은 제품을 안정적으로 만들기 위한 “공통 언어”입니다.
