# Wireframe 이해하기

# Wireframe은 무엇인가요?

### Wireframe(와이어프레임)은 화면의 구조와 정보 배치를 빠르게 설계하는 스케치입니다.

디자인을 예쁘게 만드는 것이 목적이 아니라:

* 어떤 정보가
* 어떤 우선순위로
* 어디에 배치되고
* 어떤 흐름으로 동작하는지

를 빠르게 정리하기 위한 문서입니다.

---

# 텍스트 Wireframe은 무엇인가요?

### 텍스트 Wireframe은 화면을 그림 대신 텍스트로 표현하는 방식입니다.

예시:

```text
┌─────────────────────┐
│ 상단 배너            │
├─────────────────────┤
│ 검색창               │
├─────────────────────┤
│ 추천 카드 영역       │
│ [카드1] [카드2]      │
├─────────────────────┤
│ 하단 탭바            │
└─────────────────────┘
```

빠르게 구조를 공유하고,
AI/개발/디자인 협업 속도를 높이는 데 매우 효과적입니다.

---

# 언제 / 왜 Wireframe을 쓰나요?

## 1. 아이디어를 빠르게 구조화할 때

기획 초기에는:

* 레이아웃
* 정보 우선순위
* 화면 흐름

이 중요합니다.

디자인 디테일보다는:
“무엇이 어디에 있어야 하는가”
를 먼저 정리합니다.

---

## 2. 팀 간 커뮤니케이션 속도를 높일 때

텍스트 Wireframe은:

* 기획자
* 디자이너
* 개발자
* AI 도구

모두 빠르게 이해할 수 있습니다.

특히:

* 회의
* 리뷰
* 아이디어 검토
* AI 프로토타이핑

에서 매우 효율적입니다.

---

## 3. 화면 구조를 검증할 때

Wireframe 단계에서는:

* 사용자가 이해 가능한지
* 정보량이 적절한지
* CTA가 명확한지
* 흐름이 자연스러운지

를 먼저 검토합니다.

---

# One-pager / PRD / Wireframe은 무엇이 다른가요?

| 구분     | One-pager | PRD         | Wireframe  |
| ------ | --------- | ----------- | ---------- |
| 목적     | 과제 설득     | 실제 제작 정의    | 화면 구조 설계   |
| 핵심 질문  | 왜 해야 하는가  | 무엇을 어떻게 만들까 | 어떻게 보여줄까   |
| 중심 내용  | 문제/가치     | 기능/정책/예외처리  | 레이아웃/UI 배치 |
| 상세도    | 간략        | 매우 상세       | 화면 중심 상세   |
| 주요 독자  | 의사결정자     | 기획/개발/QA    | 디자인/프론트    |
| 작성 시점  | 과제 초기     | 착수 이후       | 기능 정의 이후   |
| 문서 특징  | 논리/당위성 중심 | 정책/상태값 중심   | 시선흐름/UI 중심 |
| 대표 결과물 | 전략 요약     | 요구사항 명세     | 화면 스케치     |
| 핵심 산출  | 문제정의/Goal | 기능명세/API    | 화면배치/플로우   |

---

# 좋은 Wireframe의 특징

## 1. 정보 우선순위가 명확함

좋은 화면은:
무엇을 가장 먼저 봐야 하는지가 명확합니다.

❌

* 모든 요소 강조
* 시선 흐름 없음

✅

* 핵심 CTA 우선
* 중요한 정보 상단 배치
* 스캔 가능한 구조

---

## 2. 사용자의 행동 흐름이 자연스러움

사용자는:

* 읽고
* 이해하고
* 행동합니다.

좋은 Wireframe은:
사용자의 시선 흐름과 행동 흐름이 자연스럽게 이어집니다.

---

## 3. 실제 서비스처럼 현실적이어야 함

❌

* 너무 이상적인 구조
* 현실적 데이터 고려 없음

✅

* 실제 운영 가능한 구조
* 실제 콘텐츠 길이 고려
* 실제 에러 상황 고려

---

## 4. 예외 상태까지 고려함

좋은 Wireframe은 정상 상태만 그리지 않습니다.

예시:

* Empty State
* Loading
* Error
* Disabled
* Skeleton
* 권한 없음

등도 함께 고려합니다.

---

# Wireframe 작성 원칙

## 1. 디자인보다 구조에 집중합니다.

Wireframe의 목적은:
“예쁜 화면”이 아닙니다.

중요한 것은:

* 정보 구조
* 콘텐츠 우선순위
* 사용자 흐름

입니다.

---

## 2. 한 화면에는 한 가지 핵심 목적만 둡니다.

좋은 화면은:
무엇을 해야 하는지가 명확합니다.

예시:

* 구매
* 가입
* 탐색
* 저장
* 공유

---

## 3. 실제 콘텐츠 기준으로 작성합니다.

❌

* Lorem Ipsum
* 짧은 더미 제목만 사용

✅

* 실제 길이의 제목
* 실제 카드 개수
* 실제 에러 메시지

현실적인 콘텐츠 기준으로 설계해야 합니다.

---

## 4. 사용자의 시선 흐름을 고려합니다.

일반적으로:

* 상단 → 핵심정보
* 중앙 → 콘텐츠
* 하단 → 보조 액션

흐름으로 읽게 됩니다.

---

## 5. 상태(State)를 고려합니다.

같은 화면이라도:
상태에 따라 UI가 달라집니다.

예시:

| 상태       | 설명     |
| -------- | ------ |
| Default  | 기본 상태  |
| Loading  | 로딩 중   |
| Empty    | 데이터 없음 |
| Error    | 실패 상태  |
| Disabled | 비활성    |

---

# 텍스트 Wireframe 작성 방식

# 기본 구조

```text
┌─────────────────────┐
│ Header              │
├─────────────────────┤
│ Hero Banner         │
├─────────────────────┤
│ Main Content        │
├─────────────────────┤
│ CTA Button          │
├─────────────────────┤
│ Bottom Navigation   │
└─────────────────────┘
```

---

# 카드 구조 예시

```text
[추천 카드]

썸네일
제목
설명 텍스트 2줄
[CTA 버튼]
```

---

# 리스트 구조 예시

```text
[리스트]

- 상품명
- 가격
- 혜택
- 배송정보
```

---

# 모달 구조 예시

```text
┌──────────────┐
│ 제목         │
├──────────────┤
│ 설명 문구    │
├──────────────┤
│ 취소  확인   │
└──────────────┘
```

# 추천 작성 순서

## 1. 화면 목적 정의

예시:

* 구매 유도
* 정보 탐색
* 가입 유도
* 리텐션 강화

---

## 2. 핵심 CTA 정의

예시:

* 구매하기
* 가입하기
* 저장하기
* 공유하기

---

## 3. 필요한 정보 나열

예시:

* 상품명
* 가격
* 혜택
* 리뷰
* 추천 이유

---

## 4. 우선순위 배치

무엇을:

* 가장 먼저 보여줄지
* 가장 크게 보여줄지
* 반복 노출할지

결정합니다.

---

## 5. 예외 상태 추가

반드시:

* Empty
* Error
* Loading

상태도 함께 설계합니다.

---

# 추천 문서 분량

| 유형     | 권장 분량  |
| ------ | ------ |
| 단일 화면  | 1~3p   |
| 기능 흐름  | 5~10p  |
| 서비스 구조 | 10~30p |

---

# 추천 구성 비율

| 영역     | 권장 비중 |
| ------ | ----- |
| 화면 구조  | 50%   |
| 사용자 흐름 | 20%   |
| 상태 정의  | 15%   |
| 인터랙션   | 10%   |
| 기타     | 5%    |

---

# 마지막으로

좋은 Wireframe은
“디자인 시안 이전의 설계도”입니다.

디자인보다 중요한 것은:

* 사용자 흐름
* 정보 구조
* 행동 유도
* 상태 정의

입니다.

Wireframe 단계가 명확할수록:

* 디자인 속도
* 개발 속도
* 의사결정 품질

이 크게 좋아집니다.
